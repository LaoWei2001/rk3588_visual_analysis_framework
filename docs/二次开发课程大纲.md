# RK3588 视觉分析框架 · 二次开发循序渐进课程大纲

> **用途**：给后续的二次开发人员设计一套由浅入深、每节都能动手跑通的培训课程。
> **受众**：会 C/C++ 基础、对 OpenCV/Linux 有初步了解的开发者；不要求懂 NPU/RGA 底层。
> **如何使用**：本文是「课程设计方案」，不是逐字讲义。每节给了 目标 / 知识点 / 动手 / 验收与坑 / 对应代码与文档。
> 课程内容约 80% 已存在于本仓库的 `docs/skills/`、`src/**/README.md`、`examples/` 中——
> 备课的主要工作是**按学习顺序重新叙述 + 为每节设计练习与验收**，而不是从零写文档。

---

## 一、课程设计原则

1. **每节都有"能跑起来看见结果"的动手产出**，不空讲 API。
2. **难度单调递增、后课复用前课成果**：probe → 叠加 → ROI → 状态 → 报警 → 可配 → 上报 → 跨路 → 实战。
3. **先"只读 / 可视化"（零副作用、即时反馈、最易上手），再"有状态 / 会发出去"**。
4. **把项目里的真实坑当教学点**（坐标系 640、`static` 串台、四处对齐、ROI 不热重载、USB 偏移、`report` 字段、改了不生效）——这些是 `development_log.md` 真踩过的，最值钱。
5. **始终基于跑通的真实代码改**（照搬现有 `logic_xxx` 范式），不造玩具。
6. **课程 = 学习路径（叙事、渐进、动手）；现有 skills/README = 参考手册（查阅）**。两者分工、互相指引；每节"对应"一栏直接指回参考。

## 三、必修主线

### 模块零 · 开篇与环境

#### L0 · 课程导览 + 项目认知（1.5h）
- **目标**：建立"二次开发只碰 `src/logic`"的心智模型，知道整个系统由什么组成。
- **知识点**：
  - 这是什么（RK3588 NPU 多路视觉分析框架）、能力边界（8~15 路、约 11~13 帧）。
  - 数据流一张图：取流(`decChannel`) → RGA 缩放 → NPU 推理(`yolo`) → 跟踪(`tracker`) → **你的 logic** → 渲染(`display`) / 上报(`uploader`)。底层已封装，二次开发的"接缝"就是 logic。
  - 三种进程、三套托管：C++ 推理二进制 / Web 控制台 / 两个 Python 微服务。
  - 配置驱动思想：一台设备 = 一份 `config.json`，每路视频自由搭配 输入源 + 模型 + logic + 上报。
- **动手**：通读 `docs/development_log.md` 选段，对着架构图走查 `src/` 目录。
- **对应**：`docs/development_log.md`、`docs/skills/rk3588-console-ops/SKILL.md` §一、`docs/skills/rk3588-channel-logic/references/rk3588_yolo_系统说明文档.md`、`…/rk3588_yolo_架构图.md`。

#### L1 · 环境搭建 + 跑通第一帧（2h）
- **目标**：把环境从零跑起来、看到视频画面，体验"改 logic 名 4 秒热切换"。
- **知识点**：
  - `install_deps.sh`（国内网络已适配）；`rk3588_yolo/build.sh`（自动判别 arm/x86）；`install_app.sh`；`web_console/install.sh` → 浏览器 `:8080` 登录。
  - 用现成 `config.json` 跑一路本地视频 / RTSP，看到画面。
  - 目录地图：`src/` 各文件夹职责（capturer / analyzer / yolo / logic / config / player / uploader / core）。
- **动手**：跑通全流程 → 看到画面 → 把某通道 `logic` 改成 `logic_default` / `logic_roi`，观察热切换。
- **验收**：能独立部署、看到画面、能解释"为什么改 logic 名不用重编译"。
- **对应**：`docs/skills/rk3588-console-ops/SKILL.md` §二、各 `README.md`。

### 模块一 · 通道逻辑：读与画

#### L2 · logic 生命周期与 ChannelContext 总览（1.5h）
- **目标**：理解 logic 是"每通道、每帧"调用的纯函数，看懂 `ctx` 的角色。
- **知识点**：
  - 签名 `static void logic_xxx(ChannelContext *ctx)`；同一函数被多通道复用、靠 `ctx` 区分。
  - **坐标系统一 640 铁律**（为 L6 埋伏笔）。
  - 空指针先判空；写最小 hello logic（左上角画一行字）+ 文件末尾 `REGISTER_LOGIC` 自注册。
- **动手**：新建 `src/logic/logic_hello.cpp`，画 `"hello ch{N}"`，接到一个通道跑起来。
- **坑**：忘了 `REGISTER_LOGIC`；新增 `.cpp` 后忘了让 CMake 重新 configure。
- **对应**：`docs/skills/rk3588-channel-logic/references/channelcontext-api.md`、`docs/skills/rk3588-src-modules/logic.md` §1.1。

#### L3 · 视频帧的信息获取（1.5h）
- **目标**：吃透 `ctx->frame` 与时间相关字段。
- **知识点**：
  - `ctx->frame`（`cv::Mat`，BGR，640×640，**可能为空**）；`frame_id` 单调递增；`timestamp_ms` / `dt_ms`（计时 / 积分）；`infer_fps` / `disp_fps`；`snapshot()` 深拷贝。
  - 如何读某像素 / ROI 区域的图像数据（为选修A 传统视觉铺垫）。
- **动手**：把当前帧尺寸、`frame_id`、时间戳、实时帧率打到画面上。
- **对应**：`channelcontext-api.md` ctx 字段、`README.md` §1.2。

#### L4 · 检测结果的信息获取（通道变量调用）（2h）
- **目标**：吃透 `ctx->results` / `AlgoResult`，会用便捷查询。
- **知识点**：
  - 遍历 `ctx->results`；`AlgoResult` 全字段（`label` / `class_id` / `score` / `box` / `track_id` / `box_color` / `keypoints` / `boxMask` / `vx,vy`）。
  - **`label` 必须和 `labels.txt` 完全一致**（"一直 CLEAR"的头号原因）。
  - `box_center()` / `box_contains()` / `dist_sq_to()`；便捷方法 `has_target` / `target_count`；ROI 查询用 `roi_contains` / `roi_count_target` 等自由函数。
  - `track_id` 何时从 -1 变稳定（SORT，`tracker_min_hits`）。
- **动手**：写 `logic_probe`——把每个目标的 `label score #track_id` 标到框上方，统计画面里 person 数量。
- **验收**：能解释 `track_id = -1` 的含义；能正确按类别过滤。
- **对应**：`channelcontext-api.md` AlgoResult 节、`README.md` §1.3、`src/logic/logic_roi.cpp`。

#### L5 · 在视频上叠加图形与文字（提示文字叠加）（2h）
- **目标**：掌握 `draw_*` 四件套、中文叠加、三种绘制目标。
- **知识点**：
  - `draw_rect` / `draw_circle` / `draw_line` / `draw_text`（BGR 颜色、坐标 640）。
  - `r.box_color` 直接改框色 vs 自己画。
  - `DrawCommand::DISPLAY / UPLOAD / ALL`（屏幕看的 vs 上报图里的；状态文字常只给 DISPLAY）。
  - 中文叠加 `text_overlay`（freetype + 字体查找顺序、`text_overlay_available()`、中文 / ASCII 取舍）。
  - 图形随视频窗口缩放而变化。
- **动手**：把 L4 的 probe 升级——命中框标红、框上画 `"label score"`、左上角画**中文**状态条（"检测中 / 无目标"）。
- **坑**：上报图里没有你只画到 DISPLAY 的文字；板子缺中文字体。
- **对应**：`channelcontext-api.md` 绘制节、`README.md` §1.5、`src/player/text_overlay.h`、`src/logic/logic_person_alarm.cpp`。

### 模块二 · 区域与状态

#### L6 · ROI 区域（上）：网页绘制 + 单 ROI 判定（2h）
- **目标**：网页画 ROI → 代码读 `ctx->roi` → 点在多边形内判定。
- **知识点**：
  - 网页 ROI 节点画区 → `roi_zones.json`；**ROI 只在程序启动时加载、不热重载**（改完要停再启）。
  - `ctx->roi` 已缩放到 640；判定用 `roi_contains(ctx, box, ROI_ALL)`（没画 ROI=整帧不设限）；画 ROI 边仍用 `ctx->roi`。
  - **经典坑**：源分辨率 vs 640 坐标系；USB 摄像头 fps 绑分辨率 → ROI 偏移（固定采集分辨率解决）。
- **动手**：网页画禁区 → 只对中心落在 ROI 内的 person 标红并画区边。
- **对应**：`docs/skills/rk3588-channel-logic/references/examples/logic_roi.md`、`rk3588-console-ops` USB 偏移条目、`README.md` §1.8。

#### L7 · ROI 区域（下）：多命名区域（1.5h）
- **目标**：一个通道多个命名区域，按名字 / 序号分别处理。
- **知识点**：`ctx->rois` / `roi_count()` / `roi_at(i)` / `roi_by_name("entrance")` / `roi_contains(ctx,box,idx)` / `roi_count_target(ctx,label,roi_find(ctx,"entrance"))` / `ctx->roi_index_of(box)`；把业务步骤绑到各区域。
- **动手**：画 entrance / exit 两区，分别统计 person 数。
- **对应**：`channelcontext-api.md` 多 ROI 节、`examples/logic_multi_roi.md`、`examples/logic_wafer_sop.md`。

#### L8 · 跨帧状态与状态机（2h）
- **目标**：用 `ctx->state` 做计时 / 闩锁 / 去重，写出第一个真实报警逻辑。
- **知识点**：
  - **为何不能用 `static`（会被所有通道共享、串台）**；`ctx->state` 每通道一格 `shared_ptr` 模式。
  - **变量隔离原理**（同一函数 8 路复用为何不串——讲清这个学员才敢放心写）。
  - 计时（`timestamp_ms` / `dt_ms`）、闩锁、去重（track_id 集合）、丢检宽限。
  - 配置 vs 状态：只读参数放 `config`，运行中要改的放 `state`。
- **动手**：写"区域内有人持续 3 秒报警、离开后复位"（3 秒先写死，L9 再变可调）。
- **验收**：行为正确、不狂报、复位正常。
- **对应**：`rk3588-channel-logic/SKILL.md` 骨架、`examples/logic_hook.md`、`README.md` §1.4、`channelcontext-api.md` 变量隔离节。

### 模块三 · 配置与网页

#### L9 · 添加可调参数 + 热重载（参数添加）（2h）
- **目标**：吃透"四处对齐"，把 L8 写死的 3 秒变成可配、改完不重启即生效。
- **知识点**：
  - config registry；一条链：`config.h 字段 → REG_C → ctx->config 读 → logics.json params → 网页 LogicForm 自动渲染 → graphToConfig 落盘 → config_monitor sync_fields 热拷`。
  - **四处 key 必须完全一致**（最高频 bug）。
  - 类型表（INT / FLOAT / BOOL / STRING / STRING_ARRAY ↔ logics.json `type`，含 enum / text）。
  - **每帧现读、别缓存**（否则热重载失效）。
  - 哪些不热重载（ROI、源地址、通道数、`infer_enable`）。
  - 通道参数 vs 全局参数（`REG_C` vs `REG_G`）。
- **动手**：给停留报警加 `dwell_sec`、`cooldown_sec` 两参数，走完四处，网页改值观察 4 秒生效。
- **对应**：`docs/skills/rk3588-channel-logic/references/adding-config-parameter.md`（整篇）、`README.md` §3、`upload-and-wiring.md` §四。

#### L10 · 网页配置画布详解（web 端参数修改）（2h）
- **目标**：看懂"画布即配置"，理解参数从网页到 `config.json` 怎么流。
- **知识点**：
  - React Flow 画布与节点类型（Channel / Logic / Report / Model / ROI / Stream / Global）。
  - `logics.json` 数据驱动 `LogicForm` 自动渲染控件——**加参数前端不用改代码**。
  - `configToGraph` / `graphToConfig` 双向转换。
  - **上报节点是否生效，由 logics.json 的 `report` 字段决定**（易漏）。
  - RTSP 实时预览；复制粘贴 / 撤销快捷键。
  - 改了怎么生效（`install.sh` 重 build + 浏览器 `Ctrl+Shift+R` 硬刷新铁律）。
- **动手**：纯网页操作——拖出一路完整配置（源 + 模型 + logic + 参数 + ROI + 上报），保存后看 `config.json` 差异。
- **对应**：`docs/skills/rk3588-console-ops/references/web-console-frontend.md`、`adding-config-parameter.md` 网页机制节。

### 模块四 · 外联与跨路

#### L11 · 告警上报：业务服务器（server）（2h）
- **目标**：会用 server 上报、地址跟告警走、必做限频、上报图带框。
- **知识点**：

- **动手**：给报警逻辑接 server 上报（网页连"上报配置"节点填地址），在 `alarm_store/` 看到带地址的 `.json` 记录（Python 补传后即删）。

#### L12 · 告警上报：Dify（AI 二次分析）（1.5h）
- **目标**：会用 Dify 上报，理解"提示词人工可控"的设计。
- **知识点**：
  - `dify_uploader_enqueue`；提示词从 `ctx->config->dify_prompt` 取（网页可改、不写死在 Dify 应用里，不同通道可共用一个工作流）。
  - `event_id` 构造（`ch{N}_f{frame}_t{ts}`）。
  - 按 `track_id` 去重防刷（`logic_dify_person_verify` 范式）。
- **动手**：把 L11 改成 Dify 上报，提示词走配置，按 track_id 每个新目标只报一次。
- **对应**：`examples/logic_dify.md`、`examples/logic_dify_person_verify.md`、`development_log.md` 2026.05.08。

#### L13 · 全局逻辑（跨通道联动）（2h）
- **目标**：分清 channel vs global，写一个跨通道聚合逻辑。
- **知识点**：
  - `global_xxx` 在独立线程按 `poll_interval_ms` 轮询、一次看所有通道，**不是每帧**。
  - `get_channel_snapshot` 原子同帧取数 + `result_age_ms` 新鲜度自检；`for_each_channel`。
  - `gctx->state` 每实例一格；**不能直接往屏幕画**（只能上报或在克隆图上画）。

  - 接线：`register_global_logic` + logics.json `global_logics` + config.json 实例。
- **动手**：写"≥2 个通道同时检测到人就联动报警"，限频上报。
- **对应**：`docs/skills/rk3588-global-logic/SKILL.md`、`src/logic/global_logic.h`、`README.md` §2。

## 四、进阶选修

### 选修A · 算法范式拓展（3 课，做复杂算法才需要）
- **E1 传统视觉补位**：YOLO 检不出时用 ROI 内 HSV 占用率 / 帧差（`examples/logic_roll.md`）。
- **E2 轨迹与覆盖率**：轨迹拓宽成带 + 掩码求覆盖率（巡检 / 擦拭 / 喷涂到位，`examples/logic_wafer.md`）；动作分段横擦 / 环擦（`examples/logic_wafer2.md`）。
- **E3 姿态与 SOP**：pose 关键点做跌倒 / 挥手求救（`examples/logic_fall_detect.md`）；按顺序经过命名工位（`examples/logic_wafer_sop.md`）。

### 选修B · 平台与工程（4 课，扩展平台 / 负责部署才需要）
- **E4 部署与打包**：`install_deps.sh`、`build.sh`（arm/x86 / 98 服务器交叉编译）、`install_app.sh`、`dist/deploy.sh`、systemd 三单元。
- **E5 运维与调试**：`journalctl`、调试二分法、"改了不生效"自查、Web 终端、Redis、真实坑合集（`development_log.md` 是宝库）。
- **E6 控制台前端开发**：React + Vite、React Flow、节点、`api/client`、WebSocket、`configToGraph/graphToConfig`、install 生效铁律。
- **E7 控制台后端 + 微服务**：FastAPI 加路由（`main.py` `include_router`）、加后台服务 / 服务配置面板项、两个微服务架构（`config.yaml` / `ota_config.json`）。
  - **对应**：`web-console-frontend.md`、`services-upload-and-ota.md`、`debugging-playbook.md`。

### 选修C · 模型链路（2 课，要换 / 训练自定义模型才需要）
- **E8 自定义模型训练与转换**：`pt → onnx → rknn`、单 batch 设计、`labels.txt`、模型类型（v5 / v8 / seg / pose）如何接入。
- **E9 模型热更新 OTA**：平台远程下发、`ota_agent`、`ota_config.json` 的 `target_config` 必须等于实际在跑的配置。
