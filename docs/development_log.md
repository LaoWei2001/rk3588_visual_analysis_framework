# 开发日志（备忘录）

## 2025.12 - 2026.3

编程语言的选择：c/c++。因为底层的硬件加速依赖库只提供c++的接口，且python本身就不适合硬件性能受限的情况，多路推理远远达不到要求，使用python初步尝试多路视频推理后发现4路流连10帧都没有。还尝试过c++取流，python进行推理的方式，性能也很差。

模型结构的选择：模型输入端始终为单batch，最开始想设计为deepstream那样的多batch输入，但是性能不好,还是固定采用单batch，多线程的设计。

解决了诸多问题：如何训练出符合要求的自定义的pt模型，如何转onnx，转rknn，如何使用自己训练的模型进行图片推理。

确认程序的基本架构：各模块低耦合设计，底层解码，缩放等流程尽量与上层逻辑分离，便于二次开发。依赖配置文件设置以自由搭配各路视频的逻辑，模型，输入源类型。

## 2025.3 - 2025.4

支持平台远程下发并热更新模型文件，支持各种参数的实时更新，代码结构模块化，标准化以便于二次开发。

不断优化性能，从一开始最多支持流畅（25帧）推理2-3路视频，提高到了8路左右。

在15路并发的状态下推理帧数仍能达到每秒11-12帧，接近英伟达的jetson orin nano上使用deepstream框架推理的性能（16路13帧左右）满足简单场景的需求。

支持将视频流手动绑定至某一个npu的核心，npu有3个核心，建议总的视频流个数为3的倍数，可以使npu负载较均衡。

上报模块从主程序中解耦为python微服务程序，报警队列进入redis，后台微服务再去消费队列，c++主程序只负责将相关数据送入redis。

## 2026.05.15

如果多个通道共用同一个逻辑, 且逻辑内部用到了需要跨帧保存的变量, 需要像logic_hook和logic_dify那样使用ctx->state对跨帧状态进行维护, 否则不同通道的跨帧数据会混在一起

错误例子:

```cpp
static void logic_xxx(ChannelContext *ctx)
{
    static uint64_t last_ts = 0; // 所有通道共享，互相覆盖
    ...
}
```

正确例子:

```cpp
struct XxxState
{
    uint64_t last_ts = 0;
    bool first = true;
};
static void logic_xxx(ChannelContext *ctx)
{
    if (!ctx->state)
        ctx->state = std::make_shared<XxxState>();
    auto &s = *std::static_pointer_cast<XxxState>(ctx->state);

    // 直接用 s.last_ts、s.first，每个通道就使用的是独立的变量
}
```

## 2026.05.27

我觉得低代码有两个层次:低一点的层次本质上是一个可视化的配置工具, 配置视频源,模型,上报方式这些基本的参数以及调用开发者预先写好的算法逻辑，能够方便使用者配置参数，部署程序（粗粒度）。

高层次就支持算法编排都可以在画布上完成（细粒度）, 比如dify上可针对输入的图片利用LLM节点以及代码执行节点就可以进行自由的后续算法编排

这样的效果对于视觉分析这样定制性较高的项目来说, 完成难度极大，且大概率还是无法完成有新需求的任务，因为dify能完成相关任务我认为核心在于LLM的输出可以用自然语言定制，然后使用代码节点进行处理。

但是就目前这个多路视频分析的项目来说，实在想不到好的方法来设计代码节点然后对yolo模型输出的结果直接进行后处理（我没有办法用自然语言告诉yolo，让他输出定制化的结果），举例子：我可以在dify上让LLM直接输出图中有几个人，有没有戴安全帽，然后直接拿到结果数字做后续的算法编排。但是yolo只会输出框在哪里，类别是什么，这些都需要设计代码节点来完成，且我无法想象定制性类似那种货架区域入侵的项目怎么样直接拖拖拽拽模块就能实现。

而且就算能实现代码节点的设计，那面对一个新场景的时候也得在代码节点里面写大段的编排代码，跟我直接在代码文件里面写没有区别，而且用户使用这个低代码工具的时候也不会自己写代码。

所以我将这样低代码拖拽模块的效果以一个配置工具而不是算法编排工具来呈现。

以前新部署一个程序需要手动写json配置文件，现在可以拖拖拽拽模块然后自动生成json配置文件，我觉得也还可以，至少比之前给黄经理那种配置参数的程序强

而且现在是多路视频流，以这种配置的方式来呈现也更清晰。

## 2026.06.03

发现将usb视频源作为输入节点, 然后在ROI配置区域设定好ROI区域之后, 将底部的全局设置的最大FPS设为15, 运行时画面中的ROI没有偏差, 但是我改成25之后, 画面上的ROI就偏离了本来的位置,且视野变大了

原因是USB摄像头的帧率和采集分辨率是绑定的, 高的FPS会绑定低分辨率，此时如果在该分辨率的画布上画ROI区域用到其他分辨率上就会产生偏移。

发现问题：长时间连续15路高npu负载运行约20小时之后，cpu的负载会剧烈波动(20%~70%)，推理帧数也会剧烈波动(5~13)，且重启设备仍未改善，关闭程序一段时间后能恢复正常。

具体原因未知，很有可能是cpu温度过高触发操作系统的保护机制。

## 2026.06.12

将通道逻辑(channel_logic)做了拆分重构: 原来所有 logic_xxx 函数都挤在 channel_logic.cpp 一个文件里(2600+行), 现在每个逻辑独立成一个 src/logic/logic_xxx.cpp 文件。channel_logic.cpp 只保留框架核心(ChannelContext 方法 / draw_* / 注册分发表), 各 logic 共用的头集中到 logic_common.h(逻辑文件一行 #include 它即可)。

注册方式从「在 channel_logic_init() 里手动 register_logic」改成「文件末尾一行 REGISTER_LOGIC(...) 自注册」, 在 main() 之前的静态初始化阶段自动登记到分发表。好处: 新增一个逻辑=新增一个文件, 删除一个逻辑=删除对应文件, 互不牵连, 耦合最低。src/logic 下的 .cpp 由 CMake(aux_source_directory)自动收集编译, build.sh 每次全新构建会自动包含, 无需手动改 CMake。

注意: channel_logic_init()/deinit() 改成了空实现(仅保留以兼容 analyzer 的调用), 绝不能再在里面清零/清空注册表, 否则会抹掉静态期已完成的自注册, 导致重启后所有通道退化为空逻辑。

同步更新了 docs/skills/rk3588-channel-logic 下的全部文档(SKILL / channelcontext-api / upload-and-wiring / 系统说明文档 + 各 logic 示例)使其与新结构一致; 补齐了缺失的 logic_wafer_sop、logic_fall_detect 两个示例; 并修正了示例里过时的代码(如 logic_server/logic_dify 现已改为按 report_interval_sec 限频, 文档原来还写着无限频/写死10秒)。另外给 channelcontext-api 补上了之前完全缺失的「多 ROI(命名区域)」一节。
